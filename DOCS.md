# เอกสารระบบ (System Documentation): IoT Device Dashboard & Data Management

เอกสารฉบับนี้อธิบายสถาปัตยกรรมระบบ โครงสร้างไฟล์ และประวัติการพัฒนาของโปรเจกต์ IoT Device Dashboard ซึ่งเป็นระบบจำลองการจัดการอุปกรณ์เซนเซอร์และแสดงผลข้อมูล (Data Visualization) พัฒนาด้วย React.js, Node.js และ MongoDB โดยใช้การสื่อสารข้อมูลผ่าน gRPC 

---

## 1. วัตถุประสงค์ของโปรเจกต์ (Project Objectives)
1. **IoT Configuration:** สร้างระบบลงทะเบียนและจัดการการตั้งค่าอุปกรณ์ (Device) และเซนเซอร์ (Tags)
2. **Data Simulation:** จำลองการทำงานของเซนเซอร์แบบ Real-time ผ่าน Script Engine ฝั่ง Frontend โดยไม่ต้องพึ่งพาฮาร์ดแวร์จริง
3. **Custom Dashboard:** ให้ผู้ใช้สามารถออกแบบหน้าจอมอนิเตอร์ของตนเองได้ผ่านระบบ Drag & Drop
4. **Data Logging & Historical View:** เก็บบันทึกประวัติข้อมูลลงฐานข้อมูลและเรียกดูย้อนหลังผ่านกราฟ (Line Chart) เพื่อการวิเคราะห์ข้อมูล

## 3. คำอธิบายโครงสร้างโฟลเดอร์และไฟล์ (Directory Structure Detailed)

โปรเจกต์แบ่งออกเป็น 2 ส่วนหลัก คือ `backend` (เซิร์ฟเวอร์) และ `frontend` (หน้าเว็บ)

### 🖥️ โฟลเดอร์ `backend/` (Node.js + MongoDB + gRPC)
ทำหน้าที่เป็นศูนย์กลางเก็บข้อมูลและประมวลผลคำสั่งจาก Frontend

* **`db.js` (ไฟล์หัวใจหลักของ Backend):**
  * ทำหน้าที่เปิดการเชื่อมต่อกับ MongoDB 
  * เป็น gRPC Server ที่คอยรับคำสั่ง CRUD (Create, Read, Update, Delete) จากหน้าเว็บ
  * *จุดที่แก้ไขสำคัญ:* เพิ่มการรองรับตาราง `HistoryData` และแก้ไขบั๊กการใช้คำสั่ง `Model.create()` แทน `insertOne()`
* **`libs/schema.js`:**
  * กำหนดโครงสร้าง (Blueprint) ของตารางใน Database ประกอบด้วย:
    * `userSchema`: เก็บข้อมูลผู้ใช้งาน
    * `deviceSchema`: เก็บข้อมูลอุปกรณ์, อาร์เรย์ของ Tags, และพิกัด (X, Y) สำหรับระบบ Drag & Drop
    * `historySchema`: เก็บประวัติข้อมูล IoT รายนาที (Device ID, Date, Time, ค่าเซนเซอร์)
* **`proto/db.proto`:**
  * ไฟล์ Protocol Buffers กำหนดโครงสร้างข้อมูลและ Service สำหรับการสื่อสารแบบ gRPC 
* **`app.js` & `routes/`:**
  * ฝั่ง REST API สำหรับจัดการระบบภายนอกอื่นๆ (เช่น Auth, Preferences)
* **`common.js`:**
  * เก็บฟังก์ชันตัวช่วย (Helpers) เช่น การอ่านไฟล์ Config และการเข้ารหัสรหัสผ่าน
* **`package.json`:**
  * รายการ Dependencies ของฝั่งหลังบ้าน เช่น `mongoose`, `@grpc/grpc-js`

---

### 📱 โฟลเดอร์ `frontend/` (React.js + Material-UI)
ทำหน้าที่แสดงผล จัดการ UI/UX และจำลองการทำงานของเซนเซอร์

* **`src/index.js` และ `src/App.js`:**
  * จุดเริ่มต้นของโปรเจกต์ React ทำหน้าที่ครอบแอปพลิเคชันด้วย Context Providers ต่างๆ (Theme, Auth, Locale)
* **`src/pages/Dashboard/Device.js` (ไฟล์หัวใจหลักของ Frontend):**
  * เป็นไฟล์ที่ซับซ้อนที่สุดของโปรเจกต์ มีหน้าที่หลัก 5 ส่วน:
    1. **Device Config Form:** ฟอร์มสำหรับกรอกรายละเอียดอุปกรณ์และ Email/Line
    2. **Tag Management:** ระบบเพิ่ม/ลบ/แก้ไข Tag (เซนเซอร์) พร้อมระบบ Accordion UI
    3. **Script Engine Runner:** มีฟังก์ชัน `jsexe` และ `setInterval` (นาฬิกากลาง) ทำหน้าที่ประมวลผลสคริปต์ JavaScript จากผู้ใช้ทุกๆ 1, 15, 30 วินาที เพื่อสร้าง Live Data
    4. **Drag & Drop Canvas:** ระบบกระดานที่เปิดให้ลากกล่อง Tag, กล่อง Chart และ กล่อง Datetime จัดเรียงได้อย่างอิสระ พร้อมบันทึกพิกัด State ลง Database
    5. **Auto Data Logger & Chart:** ใช้ `recharts` สร้างกราฟ และมี Loop อีกตัวที่คอยรวบรวมข้อมูลทุก 1 นาที ยิง API ส่งกลับไปให้ `backend/db.js` บันทึกลงตาราง `HistoryData`
* **`src/pages/` (หน้าจออื่นๆ):**
  * `SignIn/`, `SignUp/`: หน้าต่างสำหรับเข้าสู่ระบบและสมัครสมาชิก
  * `Users/`: หน้าจัดการข้อมูลผู้ใช้งาน
* **`src/components/` และ `src/containers/`:**
  * เก็บชิ้นส่วน UI ที่สร้างไว้เพื่อนำไปใช้ซ้ำ (Reusable Components) เช่น แถบเมนู (Menu), แถบเลื่อน (Scrollbar), รูปแบบหน้ากระดาษ (Page Layout), และกล่องยืนยัน (QuestionDialog)
* **`src/providers/` (State Management):**
  * ใช้ React Context API จัดการสถานะระดับ Global ของแอปพลิเคชัน เช่น:
    * `Auth/`: จัดการ Token และสถานะการล็อกอิน
    * `Theme/`: จัดการโหมดสี (Dark/Light mode)
    * `Locale/`: จัดการระบบหลายภาษา
* **`src/config/`:**
  * เก็บค่าคงที่ (Constants) เช่น เส้นทาง (Routes), เมนูนำทาง (menuItems.js), และพจนานุกรมภาษา (locales)

## โครงสร้างระบบ (System Architecture)

ระบบนี้แบ่งการทำงานออกเป็น 3 ส่วนหลัก (3-Tier Architecture):

1. **Frontend (React.js):** ทำหน้าที่เป็นส่วนแสดงผล (UI) รับคำสั่งจากผู้ใช้งานและแสดงข้อมูลแบบ Real-time ผ่านการออกแบบ Dashboard
2. **Backend (Node.js):** ทำหน้าที่เป็นตัวกลาง (Middleware/Server) ประมวลผลคำสั่งจากหน้าเว็บและเชื่อมต่อกับฐานข้อมูล โดยใช้การสื่อสาร 2 รูปแบบคือ REST API และ gRPC
3. **Database (MongoDB):** ทำหน้าที่จัดเก็บข้อมูลทั้งหมดของระบบ เช่น ข้อมูลผู้ใช้, ตั้งค่าอุปกรณ์ (Device Config), และประวัติข้อมูล (History Data)

## 4.กระบวนการทำงานร่วมกันของระบบ (System Workflow: Detailed Step-by-Step)

กระบวนการทำงานของโปรเจกต์นี้ เป็นการทำงานร่วมกันระหว่าง **Frontend (React.js)**, **Backend (Node.js)**, และ **Database (MongoDB)** โดยแบ่งจังหวะการทำงานออกเป็น 6 ระยะ (Phases) ดังนี้:

### ⚙️ ระยะที่ 1: การโหลดหน้าจอและการดึงข้อมูลเริ่มต้น (Initialization & Fetching)
1. **ผู้ใช้เข้าสู่ระบบ:** เมื่อผู้ใช้เปิดเบราว์เซอร์และเข้ามาที่หน้าจัดการอุปกรณ์ (เช่น `/dashboard/12345`)
2. **React ทำงาน (Frontend):** โค้ดในไฟล์ `Device.js` จะเริ่มทำงาน (ผ่าน `useEffect`) โดยจะตรวจสอบว่า URL นี้เป็นการ "สร้างใหม่ (create)" หรือ "แก้ไขอุปกรณ์เดิม (edit)"
3. **ส่งคำขอข้อมูล (Request):** หากเป็นการแก้ไข React จะเตรียม Token ยืนยันตัวตน และยิง API ส่งคำขอไปหา Backend เพื่อขอข้อมูลอุปกรณ์รหัส `12345`
4. **Backend ประมวลผล (gRPC):** ไฟล์ `backend/db.js` จะรับคำขอผ่านฟังก์ชัน `readDocument` นำรหัสไปค้นหาในตาราง `Device` ของ MongoDB
5. **แสดงผลบนหน้าจอ:** เมื่อได้ข้อมูลกลับมา React จะเอาข้อมูลมากางลงใน State (`setDevice`) ทำให้ฟอร์มต่างๆ, รายชื่อ Tag, และตำแหน่งของกล่องบนหน้า Dashboard ปรากฏขึ้นมาตามที่ผู้ใช้เคยตั้งค่าไว้

### 🛠️ ระยะที่ 2: การตั้งค่าและการเขียนสคริปต์จำลอง (Configuration & Script Engine)
1. **ผู้ใช้เพิ่มเซนเซอร์ (Add Tag):** ผู้ใช้กดสร้าง Tag ใหม่ (เช่น Tag 1) และตั้งชื่อ Label (เช่น "Temperature")
2. **การเขียนลอจิก (Scripting):** ผู้ใช้ป้อนโค้ด JavaScript สั้นๆ ลงในช่อง Script Engine เช่น `return Math.random() * 100;`
3. **เลือกรอบเวลา (Update Interval):** ผู้ใช้ตั้งค่าว่าอยากให้เซนเซอร์นี้อัปเดตทุกๆ กี่วินาที (เช่น `1sec`, `15sec`, `1min`)
4. **การทดสอบโค้ดทันที (Test Script):** เมื่อผู้ใช้กดปุ่ม "TEST SCRIPT" ฟังก์ชัน `jsexe` (ใน Frontend) จะทำการแปลงข้อความโค้ด (String) ให้กลายเป็นฟังก์ชันจริง (Executable Function) และลองประมวลผลดู 1 ครั้ง หากสำเร็จจะแสดงผลลัพธ์สีเขียวในช่อง Console Output ทันที หากโค้ดผิดจะขึ้น Error สีแดง

### ⏱️ ระยะที่ 3: การทำงานของระบบ Real-time (Live Data Loop)
1. **นาฬิกากลางเริ่มเดิน:** ทันทีที่หน้าเว็บเปิดขึ้นมา จะมีคำสั่ง `setInterval` (นาฬิกากลางของระบบ) เดินอยู่เบื้องหลังทุกๆ 1 วินาที
2. **การตรวจสอบเงื่อนไขเวลา:** ทุกๆ 1 วินาที นาฬิกากลางจะวิ่งไปเช็ค Tag ทุกตัวว่า "ตัวไหนถึงเวลาต้องทำงานบ้าง?" (เปรียบเทียบจาก Last Run Time และ Update Interval)
   * *ตัวอย่าง:* Tag 1 ตั้งไว้ 1 วินาที (รันทุกรอบ), Tag 2 ตั้งไว้ 15 วินาที (รอให้นาฬิกากลางเดินครบ 15 รอบถึงจะยอมประมวลผล 1 ครั้ง)
3. **อัปเดตตัวเลข:** เมื่อ Tag ไหนถึงเวลารัน ระบบจะประมวลผลสคริปต์และนำตัวเลขใหม่ไปอัปเดต (Replace) ตัวเลขเก่าบนหน้าจอ
4. **วาดกราฟแบบสด (Live Chart):** ตัวเลขชุดใหม่ที่ได้ จะถูกแพ็กรวมกับ "เวลาปัจจุบัน" (เช่น `14:30:05`) และผลักเข้าไปในอาร์เรย์ `realtimeChartData` จากนั้นไลบรารี `Recharts` จะนำข้อมูล 30 วินาทีล่าสุดมาวาดเป็นเส้นกราฟที่ขยับได้แบบ Real-time ทันที

### 🖱️ ระยะที่ 4: การจัดเรียงและบันทึกแดชบอร์ด (Drag & Drop + Smart Save)
1. **การเคลื่อนย้าย (Drag):** เมื่อผู้ใช้คลิกค้างที่กล่อง Tag, Chart หรือ Datetime หน้าเว็บจะตรวจจับเมาส์ (Mouse Event) และคำนวณพิกัด X, Y แบบสดๆ ทำให้กล่องขยับตามเมาส์
2. **การปล่อยเมาส์ (Drop):** เมื่อปล่อยเมาส์ พิกัดล่าสุดจะถูกอัปเดตเข้าไปในตัวแปร State (เช่น `chartX: 450`, `chartY: 300`)
3. **การกดบันทึก (Smart Save):** เมื่อผู้ใช้กด "SAVE" ระบบจะตรวจว่าผู้ใช้ยืนอยู่หน้าต่างไหน
   * **ถ้ายืนอยู่หน้า DASHBOARD:** ระบบจะแพ็กพิกัด X, Y ที่จัดเรียงล่าสุด ส่งไปอัปเดตลง Database (ไม่งอก Tag ใหม่)
   * **ถ้ายืนอยู่หน้า DEVICES:** ระบบจะแพ็กข้อมูลการตั้งค่าทั้งหมด ส่งให้ `backend/db.js` อัปเดตตาราง `Device` ผ่านโพรโทคอล gRPC

### 💾 ระยะที่ 5: การบันทึกข้อมูลย้อนหลังอัตโนมัติ (Auto Data Logger)
1. **นาฬิกาบันทึกประวัติ (Logger Interval):** นอกจากการวาดกราฟแล้ว ยังมีนาฬิกาอีกตัวหนึ่งคอยจับเวลาแบบเงียบๆ
2. **คัดกรองข้อมูล (Filter Recordable Tags):** ทุกๆ "1 นาที" (60,000 ms) ระบบจะเช็คว่า Tag ไหนบ้างที่ผู้ใช้เปิดสวิตช์ `Record: True` เอาไว้
3. **แพ็กเกจข้อมูล (Data Packaging):** ระบบจะนำข้อมูลของ Tag ที่ต้องเรคคอร์ดมารวมกับ "วันที่" (Date) และ "เวลา" (Time) สร้างเป็นก้อนข้อมูล (JSON Object)
4. **ส่งเข้าฐานข้อมูล (Database Insertion):** Frontend จะยิง API (สร้าง Document ใหม่) ก้อนข้อมูลนี้จะถูกส่งไปที่ `backend/db.js` ซึ่งจะเรียกใช้คำสั่ง `Model.create()` เพื่อเขียนข้อมูลชิ้นนี้สลักลงไปในตาราง `HistoryData` ของ MongoDB อย่างถาวร

### 📅 ระยะที่ 6: การเรียกดูข้อมูลย้อนหลัง (Historical Data Fetching)
1. **การเปลี่ยนโหมด (Mode Switch):** เมื่อผู้ใช้คลิกที่ปฏิทิน (Datetime Widget) บนแดชบอร์ด และเลือกวันที่ย้อนหลัง (เช่น วันที่ของเมื่อวาน)
2. **เตรียมหน้าจอ:** Frontend จะรับรู้ทันทีว่าผู้ใช้ต้องการดูอดีต ระบบจะเปลี่ยนสถานะ `isHistoricalMode = true` สั่งให้กราฟหยุดวิ่งชั่วคราว และโชว์วงกลม Loading
3. **ค้นหาประวัติ (Database Query):** หน้าเว็บจะส่งคำสั่งไปยัง Backend ว่า "ขอข้อมูลทั้งหมดของ Device นี้ เฉพาะในวันที่เลือก"
4. **Backend จัดส่ง (Retrieve & Respond):** `backend/db.js` จะนำคำสั่งไปค้นใน `HistoryData` เมื่อเจอข้อมูล (อาจจะมีหลายร้อยบรรทัดที่บันทึกไว้ทุกๆ 1 นาที) จะส่งกลับมาให้ Frontend
5. **การเรนเดอร์กราฟประวัติศาสตร์ (Historical Plotting):** Frontend จะนำข้อมูลอดีตทั้งหมดมาเรียงตามเวลา (Sort by Timestamp) แล้วป้อนให้ `Recharts` วาดออกมาเป็นเส้นกราฟแบบตายตัว (Static Chart) ที่แสดงเทรนด์ของข้อมูลตลอดทั้งวันนั้น หากไม่พบข้อมูลใดๆ ระบบจะแสดงข้อความ *"ไม่มีการเก็บบันทึกข้อมูลในวันที่เลือก"* ขึ้นมาแทนอย่างชาญฉลาด
