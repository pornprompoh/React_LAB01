# เอกสารระบบ (System Documentation): IoT Device Dashboard & Data Management

เอกสารฉบับนี้อธิบายสถาปัตยกรรมระบบ โครงสร้างไฟล์ และประวัติการพัฒนาของโปรเจกต์ IoT Device Dashboard ซึ่งเป็นระบบจำลองการจัดการอุปกรณ์เซนเซอร์และแสดงผลข้อมูล (Data Visualization) พัฒนาด้วย React.js, Node.js และ MongoDB โดยใช้การสื่อสารข้อมูลผ่าน gRPC 

---

## 1. วัตถุประสงค์ของโปรเจกต์ (Project Objectives)
1. **IoT Configuration:** สร้างระบบลงทะเบียนและจัดการการตั้งค่าอุปกรณ์ (Device) และเซนเซอร์ (Tags)
2. **Data Simulation:** จำลองการทำงานของเซนเซอร์แบบ Real-time ผ่าน Script Engine ฝั่ง Frontend โดยไม่ต้องพึ่งพาฮาร์ดแวร์จริง
3. **Custom Dashboard:** ให้ผู้ใช้สามารถออกแบบหน้าจอมอนิเตอร์ของตนเองได้ผ่านระบบ Drag & Drop
4. **Data Logging & Historical View:** เก็บบันทึกประวัติข้อมูลลงฐานข้อมูลและเรียกดูย้อนหลังผ่านกราฟ (Line Chart) เพื่อการวิเคราะห์ข้อมูล

## 2. ประวัติการพัฒนาและเหตุผลการแก้ไข (3 Weeks Changelog & Reasoning)

* **เพิ่มไลบรารี `recharts` และสร้าง Mixed Chart**
  * *เหตุผล:* เพื่อเปลี่ยนข้อมูลตัวเลขดิบ (Raw Data) ให้กลายเป็นภาพกราฟเส้น (Data Visualization) ทำให้ผู้ใช้เห็นแนวโน้ม (Trend) ของเซนเซอร์ได้ชัดเจน
* **ปรับปรุงความถี่ในการอัปเดต (Update Intervals: 1sec, 15sec, 30sec, 1min)**
  * *เหตุผล:* เพื่อให้การจำลองสมจริงที่สุด เพราะในระบบ IoT จริง เซนเซอร์แต่ละตัวส่งค่าด้วยความถี่ไม่เท่ากัน การแยกเวลาประมวลผลของแต่ละ Tag ทำให้ระบบยืดหยุ่นขึ้น
* **กล่อง Datetime (ปฏิทิน) และ Historical Data Mode**
  * *เหตุผล:* แดชบอร์ดที่ดีต้องสามารถดูข้อมูลย้อนหลังได้ จึงสร้างปฏิทินแบบแยกส่วน (ลากวางได้) เพื่อให้ผู้ใช้เลือกวันที่ในอดีต และสั่งให้กราฟสลับจากโหมด Live Data ไปแสดงข้อมูลประวัติของวันนั้นๆ
* **ระบบ Auto Data Logger (Frontend Simulation)**
  * *เหตุผล:* เนื่องจากยังไม่มีระบบ Backend Worker ในเฟสนี้ จึงพัฒนาให้ Frontend ทำหน้าที่รวบรวมค่าเซนเซอร์ทุกๆ 1 นาที แล้วยิง API ไปบันทึกลง Database เพื่อให้สามารถทดสอบระบบเรียกดูข้อมูลย้อนหลังได้จริง
* **แก้ไข Backend Crash (`backend/db.js`)**
  * *เหตุผล:* พบปัญหาระบบหลังบ้านดับ (Crash) เมื่อพยายามบันทึกข้อมูลประวัติ สาเหตุจาก Dynamic Model Binding หาตาราง `HistoryData` ไม่เจอ จึงแก้ไขโดยการลงทะเบียน Model ใหม่, เปลี่ยนคำสั่งเป็น `Model.create()` และเพิ่มระบบ Safeguard (Try/Catch) ดักจับ Error

## 3. คำอธิบายโครงสร้างโฟลเดอร์และไฟล์ (Directory Structure Detailed)

โปรเจกต์แบ่งออกเป็น 2 ส่วนหลัก คือ `backend` (เซิร์ฟเวอร์) และ `frontend` (หน้าเว็บ)

### 🖥️ โฟลเดอร์ `backend/` (Node.js + MongoDB + gRPC)
ทำหน้าที่เป็นศูนย์กลางเก็บข้อมูลและประมวลผลคำสั่งจาก Frontend

* **`db.js` (ไฟล์หัวใจหลักของ Backend):**
  * ทำหน้าที่เปิดการเชื่อมต่อกับ MongoDB 
  * เป็น gRPC Server ที่คอยรับคำสั่ง CRUD (Create, Read, Update, Delete) จากหน้าเว็บ
  * *จุดที่แก้ไขสำคัญ:* เพิ่มการรองรับตาราง `HistoryData` และแก้ไขบั๊กการใช้คำสั่ง `Model.create()` แทน `insertOne()`
* **`libs/schema.js`:**
  * กำหนดโครงสร้าง (Blueprint) ของตารางใน Database ประกอบด้วย:
    * `userSchema`: เก็บข้อมูลผู้ใช้งาน
    * `deviceSchema`: เก็บข้อมูลอุปกรณ์, อาร์เรย์ของ Tags, และพิกัด (X, Y) สำหรับระบบ Drag & Drop
    * `historySchema`: เก็บประวัติข้อมูล IoT รายนาที (Device ID, Date, Time, ค่าเซนเซอร์)
* **`proto/db.proto`:**
  * ไฟล์ Protocol Buffers กำหนดโครงสร้างข้อมูลและ Service สำหรับการสื่อสารแบบ gRPC 
* **`app.js` & `routes/`:**
  * ฝั่ง REST API สำหรับจัดการระบบภายนอกอื่นๆ (เช่น Auth, Preferences)
* **`common.js`:**
  * เก็บฟังก์ชันตัวช่วย (Helpers) เช่น การอ่านไฟล์ Config และการเข้ารหัสรหัสผ่าน
* **`package.json`:**
  * รายการ Dependencies ของฝั่งหลังบ้าน เช่น `mongoose`, `@grpc/grpc-js`

---

### 📱 โฟลเดอร์ `frontend/` (React.js + Material-UI)
ทำหน้าที่แสดงผล จัดการ UI/UX และจำลองการทำงานของเซนเซอร์

* **`src/index.js` และ `src/App.js`:**
  * จุดเริ่มต้นของโปรเจกต์ React ทำหน้าที่ครอบแอปพลิเคชันด้วย Context Providers ต่างๆ (Theme, Auth, Locale)
* **`src/pages/Dashboard/Device.js` (ไฟล์หัวใจหลักของ Frontend):**
  * เป็นไฟล์ที่ซับซ้อนที่สุดของโปรเจกต์ มีหน้าที่หลัก 5 ส่วน:
    1. **Device Config Form:** ฟอร์มสำหรับกรอกรายละเอียดอุปกรณ์และ Email/Line
    2. **Tag Management:** ระบบเพิ่ม/ลบ/แก้ไข Tag (เซนเซอร์) พร้อมระบบ Accordion UI
    3. **Script Engine Runner:** มีฟังก์ชัน `jsexe` และ `setInterval` (นาฬิกากลาง) ทำหน้าที่ประมวลผลสคริปต์ JavaScript จากผู้ใช้ทุกๆ 1, 15, 30 วินาที เพื่อสร้าง Live Data
    4. **Drag & Drop Canvas:** ระบบกระดานที่เปิดให้ลากกล่อง Tag, กล่อง Chart และ กล่อง Datetime จัดเรียงได้อย่างอิสระ พร้อมบันทึกพิกัด State ลง Database
    5. **Auto Data Logger & Chart:** ใช้ `recharts` สร้างกราฟ และมี Loop อีกตัวที่คอยรวบรวมข้อมูลทุก 1 นาที ยิง API ส่งกลับไปให้ `backend/db.js` บันทึกลงตาราง `HistoryData`
* **`src/pages/` (หน้าจออื่นๆ):**
  * `SignIn/`, `SignUp/`: หน้าต่างสำหรับเข้าสู่ระบบและสมัครสมาชิก
  * `Users/`: หน้าจัดการข้อมูลผู้ใช้งาน
* **`src/components/` และ `src/containers/`:**
  * เก็บชิ้นส่วน UI ที่สร้างไว้เพื่อนำไปใช้ซ้ำ (Reusable Components) เช่น แถบเมนู (Menu), แถบเลื่อน (Scrollbar), รูปแบบหน้ากระดาษ (Page Layout), และกล่องยืนยัน (QuestionDialog)
* **`src/providers/` (State Management):**
  * ใช้ React Context API จัดการสถานะระดับ Global ของแอปพลิเคชัน เช่น:
    * `Auth/`: จัดการ Token และสถานะการล็อกอิน
    * `Theme/`: จัดการโหมดสี (Dark/Light mode)
    * `Locale/`: จัดการระบบหลายภาษา
* **`src/config/`:**
  * เก็บค่าคงที่ (Constants) เช่น เส้นทาง (Routes), เมนูนำทาง (menuItems.js), และพจนานุกรมภาษา (locales)

## 4. กระบวนการทำงานร่วมกันของระบบ (System Workflow)

1. **การสร้างเซนเซอร์จำลอง:** ผู้ใช้เปิดหน้า Dashboard -> สร้าง Tag ใหม่ -> ใส่สคริปต์สุ่มตัวเลข (เช่น `return Math.random() * 100`) -> กด Save
2. **การบันทึกข้อมูล:** `Device.js` (Frontend) จะนำโครงสร้าง Tag และพิกัดหน้าจอ ส่งผ่าน gRPC ไปยัง `db.js` (Backend) เพื่ออัปเดตลงตาราง `Device` ใน MongoDB
3. **การทำงานของ Live Data:** นาฬิกากลางใน React จะสั่งรันสคริปต์ตามเวลาที่ตั้งไว้ (เช่น ทุก 1 วิ) และเอาตัวเลขไปโชว์ในกล่องข้อความพร้อมกับวาดลงบน `LineChart`
4. **การเก็บประวัติ (Data Logging):** หากเปิดโหมด "Record" หน้าเว็บจะแพ็คข้อมูลชุดล่าสุดทุกๆ 1 นาที แล้วส่งให้ `db.js` สร้างเป็น Document ใหม่ลงในตาราง `HistoryData`
5. **การเรียกดูย้อนหลัง:** ผู้ใช้คลิกปฏิทินเลือกวันในอดีต กราฟจะแสดงวงกลม Loading และยิงคำสั่งค้นหาข้อมูลของวันนั้นจากตาราง `HistoryData` นำมาวาดเป็นกราฟประวัติศาสตร์ 24 ชั่วโมง
